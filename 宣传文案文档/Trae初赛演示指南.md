# Trae 初赛 · 演示操作指南

> **知墟 (ZhiXu) — AI Code Copilot 观测与进化平台**
> GitHub: https://github.com/MarioM2026/zhixu-ACOP-AICodeCopilot
>
> **演示时长**：5 ~ 8 分钟
> **演示目标**：展示知墟如何将 Trae IDE 的 AI 调用从"黑盒"变为"全透明"，并通过规则引擎实现自动优化

---

## 一、演示前准备（1 分钟）

### 1.1 启动服务

```bash
# 终端 1：启动后端（默认 3001 端口）
npm run dev:server

# 终端 2：启动前端（默认 3000 端口）
npm run dev:client
```

### 1.2 打开浏览器

- 前端看板 → http://localhost:3000
- 后端 API → http://localhost:3001/api/events

### 1.3 连接 Trae 数据源

1. 打开浏览器访问 **http://localhost:3000**
2. 点击顶部导航栏 **「设置」**
3. 在「Trae 日志目录」输入框填入（Windows 默认路径）：

```
%APPDATA%/TRAE SOLO CN/logs
```

> 💡 如果你用的是 Trae 其他版本，路径可能是：
> - `%APPDATA%/Trae/logs`
> - `C:\Users\<用户名>\AppData\Roaming\TRAE SOLO CN\logs`

4. 点击 **「保存设置」**，后端会自动扫描 Trae 日志
5. 等待 3~5 秒，回到 **「仪表盘」** 页面，观察数据是否开始刷新

### 1.4 预热数据（可选）

如果没有现成的 Trae 日志，可以手动触发几次 Trae AI 对话（让 Trae 帮你写代码），再回到仪表盘，数据就会自动出现。

---

> **演示者说：**

> "我叫 XXX，今天演示的项目叫『知墟』。它解决的是一个很具体的问题——你用 Trae、Cursor 这些 AI 编程助手，每个月花了多少 Token？哪类任务响应最慢？上下文什么时候该清理了？——这些在以前都是黑盒。

---

### 环节 2：仪表盘 — 全局数据一目了然（1.5 分钟）

**操作步骤：**

1. 确保在 **http://localhost:3000** 首页（仪表盘）
2. 观察顶部 **4 个核心指标卡片**：
   - 💰 **总 Token 消耗**（今日 / 近 7 天）
   - 📊 **总请求次数**
   - ⏱️ **平均响应延迟**
   - ❌ **错误率**

> **演示者说：**
>
> "这是仪表盘，开屏即见。4 个核心指标一目了然——不用去翻日志，不用算表格。
>
> 这张图展示了近 7 天的 Token 消耗趋势，蓝色是输入 Token，绿色是输出 Token。哪一天用量最高、哪个时段最耗 Token，一眼就能看出来。"

3. 向下滚动，观察 **「工具使用分布」饼图**：
   - 显示 Trae / Cursor / Claude Code 各占多少

> "我们支持多工具统一采集。可以看到目前主要是 Trae 的数据，这是模型使用分布。"

---

### 环节 3：事件流 — 每次 AI 调用都有记录（1.5 分钟）

**操作步骤：**

1. 点击顶部导航 **「事件」**
2. 观察事件列表：
   - 每行代表一次 AI 调用
   - 显示：时间戳、模型名、输入/输出 Token、延迟

> **演示者说：**
>
> "事件流是知墟的核心。每次 AI 调用都留下记录。
>
> 可以看到每次调用的：具体用的是什么模型、Doubao 还是 Qwen、Token 消耗了多少、响应花了多少毫秒。
>
> 这解决了一个实际问题：很多团队不知道 AI 编程助手在哪些任务上消耗最多 Token、哪些模型响应最快——现在都透明了。"

3. 点击任意一行事件，展开查看详情（会话 ID、trace ID）

> "点开一行可以看到完整的调用上下文信息，包括 session ID 和 trace ID，方便追溯。"

---

### 环节 4：上下文管理 — 给会话打分（1 分钟）

**操作步骤：**

1. 点击顶部导航 **「上下文」**
2. 观察会话列表，每个会话右侧有 **「健康评分」**

> **演示者说：**
>
> "上下文管理是知墟的第二个核心功能。
>
> 每个会话都有一个健康评分，基于 4 个因子：时效性、Token 量、代码接受率、任务复杂度。
>
> 评分高的建议保留，评分低的——比如这个会话已经 60k token 了还频繁报错——系统会建议清理或新建会话。
>
> 以前开发者靠感觉决定什么时候清理上下文，现在有数据支撑。"

---

### 环节 5：规则引擎 — 从监控到自动进化（1.5 分钟）

**操作步骤：**

1. 点击顶部导航 **「规则」**
2. 观察预置规则列表

> **演示者说：**
>
> "规则引擎是知墟的灵魂。它把被动监控升级为主动进化。
>
> 我们内置了 5 条开箱即用的规则：
>
> ① Token 消耗超限告警 —— 比如设定日消耗超过 5 万 token 就触发钉钉通知
> ② 错误率阈值告警 —— 10 分钟内错误超过 3 次就告警
> ③ 上下文 Token 占用超 80% —— 自动建议清理
> ④ 高频语法错误 —— 自动注入优化提示
> ⑤ 延迟异常 —— 响应超过 3 秒就通知
>
> 所有规则都是 YAML 配置，无需改代码，告警通道支持钉钉、邮件、Webhook。"

3. 展示 **「添加规则」** 按钮（演示可点开看一下）

> "规则可以随时新增。比如我想加一条：TTFT（首 Token 时间）超过 2 秒就注入并发调用提示——点添加规则，填条件、配动作，保存即可。"

> "最后总结一下知墟的技术亮点：
>
> 1. **多工具统一适配器**：Trae / Cursor / Claude Code 三端统一采集，数据模型一致

## 四、Q&A 准备

| 问题 | 回答要点 |
|------|----------|
| Q: 和现有监控工具的区别？ | 不是通用监控，是专为 AI 编程助手设计的，模型名、Token、TTFT、上下文健康度都是 AI 编程特有的指标 |
| Q: 支持哪些模型？ | 支持所有在日志里有记录的模型，包括 Doubao、Qwen、Claude 等，通过模型名提取自动识别 |
| Q: 数据存在哪里？ | 本地 JSON 文件存储，可配置数据目录，不上传到任何第三方 |
| Q: 规则引擎能做什么动作？ | 目前支持：告警通知（钉钉/邮件/Webhook）、标记会话为「建议清理」、自动注入 Prompt |
| Q: 有 API 吗？ | 有 REST API，可以查询事件、触发规则、注入 Prompt，方便集成到自己的流程里 |
| Q: 未来规划？ | v2.0 支持模型自动路由、插件系统，v3.0 实现完整的反馈闭环 |
